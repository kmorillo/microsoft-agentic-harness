Chapter 02 · Infrastructure

Compute & AI Services

Where the harness actually runs, and how Azure OpenAI powers the agent. This page covers hosting options, model deployment, and the configuration that connects code to cloud.

Hosting options

The harness is a standard ASP.NET Core application, so it can run on any Azure compute service that supports .NET 10 containers. The choice comes down to how much operational overhead you want to take on, what scaling model fits your workload, and whether you need Docker-in-Docker for the sandbox executor.

Factor Container Apps App Service AKS
Best for Most deployments Simple / single-app Enterprise at scale
Scaling Event-driven, 0 → N Fixed / auto rules Full Kubernetes
Cost model Consumption or dedicated Always-on tiers Node pools
SignalR Built-in WebSocket Needs ARR affinity Ingress controller
Sandbox (Docker.DotNet) Needs dedicated plan Not supported Native Docker socket
Complexity Low–Medium Low High
✓
Our recommendation

Container Apps for most teams. It handles the SignalR WebSocket requirement natively, scales to zero in dev, and supports the dedicated compute plan needed for Docker.DotNet sandbox isolation.

Container Apps layout

The harness deploys as three Container Apps, each with a distinct responsibility. Splitting them lets you scale the API and SignalR hub independently and isolate the MCP server for external consumers.

Container Apps Environment
⚙
harness-api
Main ASP.NET Core API. Hosts the agent runtime, MediatR pipeline, CQRS handlers.
Ingress: external, port 8080
⚙
harness-hub
AgentHub SignalR server. Real-time agent-to-UI streaming. Needs sticky sessions.
Ingress: external, port 8081
⚙
harness-mcp
MCP Server. Exposes tools/prompts/resources via Model Context Protocol over HTTP. JWT-authenticated.
Ingress: internal or external

Bicep example

A representative Container App resource definition for the main API. Note how secrets reference Key Vault via managed identity rather than embedding values.

bicep 
resource harnessApi 'Microsoft.App/containerApps@2024-03-01' = {
  name: 'harness-api'
  location: location
  properties: {
    environmentId: containerAppEnv.id
    configuration: {
      ingress: {
        external: true
        targetPort: 8080
        transport: 'auto'
      }
      secrets: [
        {
          name: 'openai-key'
          keyVaultUrl: '${keyVault.properties.vaultUri}secrets/AzureOpenAI-ApiKey'
          identity: managedIdentity.id
        }
      ]
    }
    template: {
      containers: [
        {
          name: 'harness-api'
          image: '${containerRegistry}/harness-api:${imageTag}'
          resources: { cpu: json('1.0'), memory: '2Gi' }
          env: [
            {
              name: 'AI__AgentFramework__AzureOpenAI__Endpoint'
              value: openAiEndpoint
            }
            {
              name: 'AI__AgentFramework__AzureOpenAI__ApiKey'
              secretRef: 'openai-key'
            }
          ]
        }
      ]
      scale: {
        minReplicas: 1
        maxReplicas: 10
        rules: [
          {
            name: 'http-rule'
            http: { metadata: { concurrentRequests: '50' } }
          }
        ]
      }
    }
  }
}

Azure OpenAI

The harness requires two model deployments in Azure OpenAI. They can live on the same resource or on separate resources if you need to isolate quota.

Agent model

A chat completion model powers the agent runtime. It handles conversation, tool selection, and reasoning. This is the model that decides when to call tools, how to interpret results, and what to say to the user. Configure via AppConfig.AI.AgentFramework.AzureOpenAI.

Embedding model

An embedding model generates vector representations for document chunks and queries in the RAG pipeline. Used during both ingestion (indexing documents) and retrieval (embedding the user's query for similarity search). Configure the deployment name and dimensions via AppConfig.AI.Rag.Embeddings.

Configuration mapping

Each code-level config path maps to a specific Azure resource or setting. Use this table to trace from appsettings.json to the Azure portal.

Code Config Path Azure Resource What It Controls
AI:AgentFramework:AzureOpenAI:Endpoint Azure OpenAI resource endpoint Agent LLM calls
AI:AgentFramework:AzureOpenAI:DeploymentName Chat completion deployment Model selection
AI:AgentFramework:AzureOpenAI:ApiKey Azure OpenAI API key (or use managed identity) Authentication
AI:Rag:Embeddings:Endpoint Same or separate OpenAI resource Embedding generation
AI:Rag:Embeddings:DeploymentName Embedding deployment Embedding model
AI:Rag:Embeddings:Dimensions N/A (client-side config) Vector dimensions (1536/3072)
!
Token limits matter

Most chat models have large context windows, but the harness uses a context budget system (see the Developer Guide) to stay well under. Plan your Azure OpenAI quota (TPM — Tokens Per Minute) based on expected concurrent users. A single agent turn can consume 4K–20K tokens depending on tool use and RAG context.

AI Foundry integration

Azure AI Foundry (formerly Azure AI Studio) provides a management layer for model deployments, evaluations, and prompt flows. The harness connects to it via the standard Azure OpenAI endpoint — AI Foundry adds governance (content filters, usage tracking) on top. If you are using Anthropic models via AI Foundry, the endpoint shape changes slightly (see the AppConfig.AI.AgentFramework.Anthropic configuration section).

i
Prefer managed identity

Managed Identity is preferred over API keys for production. Set AI:AgentFramework:AzureOpenAI:UseManagedIdentity: true and assign the Cognitive Services OpenAI User role to your Container App's managed identity.
