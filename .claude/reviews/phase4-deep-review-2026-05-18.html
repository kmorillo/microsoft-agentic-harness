Critical Findings

CriticalInfrastructure
C-1: CancelAsync and RetryStepAsync are silent no-ops returning Success
Infrastructure.AI/Planner/PlanExecutor.cs:71-81
Both methods log a message and return Result.Success() without doing anything. Callers (CQRS handlers) believe the operation succeeded. A user invoking Cancel on a running plan would receive a success response while the plan continues executing.
Fix: Return Result.Fail("Not yet implemented") or implement actual cancellation via the per-plan CancellationTokenSource.
CriticalInfrastructure
C-2: Double version increment — interceptor + manual entity.Version++
Infrastructure.AI/Planner/EfCorePlanStateStore.cs:143,261,295 + Persistence/SqliteVersionInterceptor.cs:40-57
EfCorePlanStateStore manually increments entity.Version++ in 3 places. SqliteVersionInterceptor also increments Version on all modified entities during SavingChanges. Every save increments by 2, breaking optimistic concurrency — the WHERE clause uses the original value, causing silent version drift.
Fix: Remove all entity.Version++ lines from EfCorePlanStateStore. Let the interceptor handle it exclusively.
CriticalInfrastructure
C-3: AllowedExpressionRegex contradicts IsExpressionSafe dot rejection
Infrastructure.AI/Planner/StepExecutors/ConditionalBranchStepExecutor.cs:72-84, 237
Line 77 rejects expressions containing ., but AllowedExpressionRegex at line 237 includes \. in its character class. The UnsafeExpressionRegex checks for patterns like System\. which can never match because dot rejection happens first — making the entire unsafe pattern check dead code. The security boundary is incoherent.
Fix: Remove \. from AllowedExpressionRegex. Remove System\. etc. from UnsafeExpressionRegex (dead code). Or restructure so safety checks are coherent.
CriticalSecurity
C-4: Operator precedence bug in conditional evaluator
Infrastructure.AI/Planner/StepExecutors/ConditionalBranchStepExecutor.cs:120-140
Code splits on && first (calling it "lower precedence") then ||. But && has HIGHER precedence than ||. Expression a || b && c evaluates as (a || b) && c instead of a || (b && c). This will produce wrong branch decisions in plans with compound conditions.
Fix: Swap: split on || first (lower precedence, .Any()), then && (higher precedence, .All()).
CriticalSecurity
C-5: Command injection via unsanitized Arguments in ProcessSandboxExecutor
Infrastructure.AI/Sandbox/ProcessSandboxExecutor.cs:118-119
AllowedPrograms check validates the executable name but request.Arguments is passed directly to ProcessStartInfo.Arguments without sanitization. Shell metacharacters (&, |, ;) in Arguments can inject arbitrary commands. Same class of issue in DockerSandboxExecutor.cs:153-157 where Arguments is passed as a single Docker Cmd entry.
Fix: Switch to ProcessStartInfo.ArgumentList (separate entries, no shell interpretation). Change SandboxExecutionRequest.Arguments from string? to IReadOnlyList<string>?.
CriticalSecurity
C-6: AllowedPrograms bypass — open by default when list is empty
Infrastructure.AI/Sandbox/ProcessSandboxExecutor.cs:107-109
When AllowedPrograms is empty (the default), the check is skipped entirely — any executable can launch. For a security boundary, this should be closed-by-default: empty allowlist = refuse execution.
Fix: If AllowedPrograms.Count == 0, throw UnauthorizedAccessException.
CriticalTests
C-7: No tests for command injection, sandbox bypass, or HMAC weak keys
Multiple test files
Security-critical paths have zero adversarial test coverage:

  • No test for sandbox-disabled capability bypass
  • No HMAC weak key rejection test (key < 32 bytes)

  • No Docker container escape vector tests (--privileged flag, PID namespace)

CriticalTests
C-8: Retry/backoff system entirely untested at integration level
Infrastructure.AI.Tests/Planner/PlanExecutorTests.cs
Every PlanExecutor test uses RetryPolicy { MaxRetries = 0 }. Zero testing of: steps that fail and retry, exponential backoff delays, OnExhausted = SkipStep behavior, or AttemptCount incrementing. The entire retry subsystem is untested.
Fix: Add tests with MaxRetries > 0, a step executor that fails N times then succeeds, and verify attempt counts + delays.

Strengths

What's Done Well

  • Immutability: Consistent sealed record usage across all domain models. IReadOnlyList<T> on all non-config public surfaces.
  • XML Documentation: Comprehensive across nearly all public types and members — excellent for a template.
  • Docker Hardening: CapDrop: ALL, no-new-privileges, ReadonlyRootfs, User: 65534:65534, network isolation. Textbook container security.
  • HMAC Implementation: HMACSHA256.HashData (static), FixedTimeEquals for timing-safe comparison, ZeroMemory for cleanup. Versioned key rotation.
  • Security Model: Deny-overrides-allow in CapabilityEnforcer. Path normalization with traversal detection. Host matching with port stripping.
  • Clean CQRS: Consistent command/query/handler/validator triplets. Proper Result pattern throughout.
  • Strongly-typed IDs: PlanId and PlanStepId prevent primitive obsession.
  • EF Core: Proper IDbContextFactory for short-lived contexts. Good entity configurations with value converters.
  • Keyed DI: Step executors registered with StepType enum keys, following established project patterns.
  • Test Volume: ~5,600 lines of tests across 37 files covering most happy paths and basic error cases.
